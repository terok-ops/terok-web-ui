# Terok Web UI: Agent + Local Web UI

This setup runs an **AI agent inside a Podman container** that also hosts a single project workspace. The pattern is:

* **One project per container**, mounted at:
  `/workspace/<projectname>`
* **The agent** operates inside that workspace directory, with shared auth/config in `~/.codex`.
* You can drive the agent **from the CLI** (interactive or `exec`) **or via the SDK** (programmatic, streaming events).
* The files in `/opt/terok-web-ui` provide a **lightweight web UI** (SDK-powered) that mimics a "Cloud AI" flow:
  discuss code, inspect thinking/command execution, and **export/apply diffs**.

---

## Container Layout

* **Project root**: `/workspace/<projectname>`
  The UI treats this as its repository root (set by `REPO_ROOT` env).
* **Terok Web UI** (this app): `/opt/terok-web-ui` — the UI is meant to be installed there inside the container.

---

## How the Local UI Works

* The **browser** hits the Express server in this container.
* The server uses **`@openai/codex-sdk`** to start a **streamed run** bound to `REPO_ROOT`.
* It forwards **structured events** over **SSE** to the browser:

  * `thinking` — model's internal reasoning (rendered collapsible)
  * `message` — final/user-facing content (Markdown)
  * `tool.*` — commands the agent runs (shown live in a "Commands" tab)
  * `diff` — unified patches the agent proposes (shown in a "Diffs" tab)
  * `done` — end of turn
* The UI can **apply the last diff** via the backend using `git apply --index`.

This mirrors the Cloud UI workflow, but the agent edits **your local workspace** directly.

## Repository layout

* **Project root** – `/opt/terok-web-ui` inside the production image (or `/workspace/codex-in-podman` when iterating locally)
  * `server.js` — Express server that bridges the browser UI with `@openai/codex-sdk` threads. It streams events, captures
    command output, stores the last diff, exposes file-browser APIs, and wires in model selection.
  * `lib/` — runtime helpers used by the server:
    * `config.js` — derives repo paths, environment defaults, and shared config such as `REPO_ROOT`, logging destinations, and
      fallback model lists. Also provides `resolveRepoPath` to guard file access.
    * `auth.js` — loads API keys from env or `~/.codex/auth.json`.
    * `runStore.js` — in-memory store keyed by run id (prompt, command log, last diff).
    * `logging.js` — appends structured lines to `/opt/terok-web-ui/terok-web-ui.log` (configurable via `TEROK_LOG`).
    * `models.js` — tracks the active model/effort, fetches model ids from the OpenAI API, and merges with local fallbacks.
  * `static/` — framework-free browser UI served from `/static/`.
    * `index.html` — chat pane, command log, diff viewer, file browser, and model/effort selectors backed by `/api/model`.
    * `marked.min.js` — Markdown renderer for agent responses.
  * `package.json` — Node ESM project (`express` + `@openai/codex-sdk`).
  * `dist/` — compiled server output (generated by CI/build; not checked in).
  * `node_modules/` — installed dependencies (not checked in; installed in CI/runtime).

The Express server is configured to serve static assets from `$TEROK_DIR/static` in production, but during development the
repo's `static/` directory mirrors that layout.

## Container bootstrap

The `uc.terok-web-ui.sdk.containerfile` image (see sample snippet below) wires up the runtime like so:

1. Verifies SSH material in `/home/dev/.ssh`, copies it to `/root/.ssh`, and warms `github.com` host keys.
2. Clones (or hard-resets) `TEROK_REPO` into `TEROK_DIR` – defaults to `/opt/terok-web-ui`.
3. Runs `npm ci --omit=dev` (or `npm install --omit=dev`) inside `TEROK_DIR`.
4. Clones (or fetches) the workspace project repo into `REPO_ROOT` – defaults to `/workspace/ultimate-container`.
5. Executes `node server.js`, which binds to `$HOST:$PORT` (defaults `0.0.0.0:7860`).

```
ENV REPO_ROOT="/workspace/ultimate-container" \
    CODE_REPO="https://gitlab.com/electric-sheep/ultimate-container.git" \
    HOST="0.0.0.0" PORT="7860" \
    TEROK_DIR="/opt/terok-web-ui" \
    TEROK_REPO="git@github.com:terok-ops/terok-web-ui.git"
```

## Release artifacts

On merges to `main`/`master`, CI builds a production bundle and publishes a
`terok-web-ui-dist.tar.gz` asset in the `latest` GitHub release. The tarball contains:

* `dist/` — compiled server output plus copied static assets.
* `node_modules/` — production dependencies.
* `package.json` — metadata for runtime inspection.

Container entrypoints can download and extract this asset to avoid a git clone
and `npm install` on startup.

## Runtime behaviour

* **Runs & streaming**
  * `/api/send` creates a run id via `runStore.createRun` and logs the prompt.
  * `/api/stream/:id` starts an agent thread in `REPO_ROOT` (default `/workspace`) and forwards streamed events to the browser
    (thinking, messages, command execution, file diffs, status updates). Diffs are cached via `setLastDiff` so they can be
    applied later.
  * `/api/cmd-log/:id` and `/api/last-diff/:id` expose the captured command log and the most recent diff.
* **Diff application**
  * `/api/apply/:id` writes the cached diff to a temp file inside `REPO_ROOT` and shells out to `git apply --index`.
* **Repository browser**
  * `/api/list`, `/api/read`, `/api/save` all go through `resolveRepoPath` to prevent path escapes and implement the UI's
    simple file tree + inline editor.
* **Model management**
  * `/api/model` (GET/POST) surfaces the current model/effort along with defaults pulled from `~/.codex/config.toml` (or
    env overrides). `models.js` keeps a cached list fetched from the OpenAI API.
* **Logging**
  * Every run/action is logged via `lib/logging.js` to stdout and (when available) `/opt/terok-web-ui/terok-web-ui.log`.

## Environment knobs

These env vars influence behaviour (all optional):

| Variable | Purpose |
| --- | --- |
| `REPO_ROOT` | Working directory passed to agent threads (default `/workspace`). |
| `HOST` / `PORT` | Express bind address/port (defaults `0.0.0.0:7860`). |
| `TEROK_LOG` | Override the log file path (defaults to `/opt/terok-web-ui/terok-web-ui.log`). |
| `TEROK_BACKEND` | Select the active backend (`codex`, `claude`, or `mistral`). |
| `TEROK_MODEL`, `TEROK_EFFORT` | Force default model/effort without editing `~/.codex/config.toml`. |
| `TEROK_CLAUDE_MODEL` | Override the default Claude model when using the Claude backend. |
| `TEROK_MISTRAL_MODEL` | Override the default Mistral model when using the Mistral backend. |
| `TEROK_SKIP_GIT_CHECK` | Skip verifying that `REPO_ROOT` contains a Git repo (enabled automatically if `.git` missing). |
| `TEROK_MODEL_CACHE_MS` | TTL for cached model lists (defaults to 5 minutes). |
| `TEROK_MODEL_FETCH_TIMEOUT_MS` | Timeout for the model list request (defaults to 5 seconds). |

The app reads OpenAI API keys from `OPENAI_API_KEY`, `CODEX_API_KEY`, or `~/.codex/auth.json`. For Claude, set
`TEROK_CLAUDE_API_KEY`, `ANTHROPIC_API_KEY`, or `CLAUDE_API_KEY`. For Mistral, set `TEROK_MISTRAL_API_KEY`
or `MISTRAL_API_KEY`.

## Typical workflow

1. Mount a project at `/workspace/<name>` and set `REPO_ROOT` accordingly if needed.
2. Ensure `~/.codex/config.toml` and `~/.codex/auth.json` exist (shared with the CLI).
3. Start the server (`node server.js`) and open `http://localhost:7860`.
4. Chat with the agent, inspect streamed events (thinking/commands/diffs), tweak the active model/effort, apply diffs, and use
   the file browser to inspect or edit files in-place.

For aspirational UI ideas, see `ROADMAP.md`.

This document reflects the current state of the repo so agents and humans know where logic lives and how to extend it safely.
